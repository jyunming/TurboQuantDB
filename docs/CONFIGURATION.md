# TurboQuantDB ‚Äî Configuration Guide

This guide covers every configuration dimension exposed by `Database.open()` and `db.search()`, how they interact, and which presets to use for common scenarios.

For full empirical numbers across 32 configs √ó 4 datasets (GloVe-200, arXiv-768, DBpedia-1536, DBpedia-3072) see [`_full_config_report.md`](../benchmarks/_full_config_report.md) (generated by `python benchmarks/full_config_bench.py --full`).

---

## Parameter Quick Reference

```python
db = Database.open(
    path,
    dimension,
    bits=4,                 # 1..8; bits=1 requires fast_mode=True. Common: 2, 4, 8.
    rerank=False,           # False (default) = MSE codes only, minimum disk
                            # True = store raw INT8 vectors for exact second-pass rescoring
    rerank_precision=None,  # None (‚Üí "int8"), "int8"/"i8", "residual_int4"/"ri4",
                            # "int4"/"i4" (deprecated), "f16", "f32", "disabled"
    fast_mode=True,         # True (default) = MSE-only (fastest ingest, minimum disk)
                            # False = MSE + QJL residual (+5‚Äì10 pp R@1 at d ‚â• 1536, rerank=False)
    quantizer_type=None,    # None auto-selects dense for d<1024, srht for d>=1024
    metric="ip",            # "ip" (inner product), "cosine", or "l2" (Euclidean)
    seed=42,
)

results = db.search(
    query,
    top_k,
    rerank_factor=None,     # int multiplier for rerank oversampling; default 10 (brute) / 20 (ANN)
    _use_ann=None,          # None (auto), True (force HNSW), False (force brute)
)
```

## Dimension-by-Dimension Guide

### `bits` ‚Äî 2 or 4

Controls the number of bits per codebook subspace (Lloyd-Max quantization).

| bits | Compressed size | Recall cost | When to use |
|------|----------------|-------------|-------------|
| 4    | ~0.5 bytes/dim | baseline    | Almost always |
| 2    | ~0.25 bytes/dim | ‚àí5 to ‚àí15 pp R@1 | Storage extremely constrained |

**Rule:** Use `bits=4`. The compressed size advantage of `bits=2` is modest (2√ó over `bits=4`) while the recall penalty is large. At `rerank=True`, the raw-vector storage dominates disk anyway, making the code size difference irrelevant.

---

### `fast_mode` ‚Äî True (default, MSE-only) or False (MSE+QJL)

Controls whether 1-bit QJL residual codes are stored alongside the MSE codes.

| fast_mode | Ingest cost | Disk overhead | Recall effect |
|-----------|-------------|---------------|---------------|
| True **(default)** | baseline | 0 | baseline (all bits to MSE codebook) |
| False     | +20‚Äì40%     | +4‚Äì8 MB/100k vectors | d < 512: **‚àí2 to ‚àí8 pp R@1** (noise) |
|           |             |               | d ‚â• 1536: **+5‚Äì10 pp R@1** (signal accumulates) |

**Rule:**
- `d < 512` ‚Üí `fast_mode=True` always. QJL projections at low d are too noisy and hurt recall.
- `d ‚â• 1536` ‚Üí `fast_mode=False` when `rerank=False`. QJL residuals add useful signal.
- `rerank=True` ‚Üí `fast_mode` is secondary. Float precision re-scoring dominates regardless of residual quality.

---

### `rerank` + `rerank_precision` ‚Äî exact rescoring

When `rerank=True`, raw vectors are stored on disk alongside compressed codes. During search, the top `rerank_factor √ó top_k` candidates from the quantized pass are re-scored with exact dot products against the stored raw vectors.

| rerank_precision | Storage per vector | Precision | When to use |
|------------------|--------------------|-----------|-------------|
| `"int8"` / `"i8"` **(default)** | d+4 bytes | ~0.5% error at unit norm | Default ‚Äî 2√ó smaller than F16, same recall |
| `"int4"` / `"i4"` | ‚åàd/2‚åâ+4 bytes | ~8% error | Tight storage budget; minor recall trade-off |
| `"f16"` | 2 bytes/dim | ~0.001 error | Maximum precision, storage not critical |
| `"f32"` | 4 bytes/dim | exact | Full precision; rarely needed |
| `"disabled"` / `"dequant"` | 0 | zero gain for IP | No raw storage; kept for legacy |

**Default:** `rerank_precision=None` ‚Üí `"int8"` (INT8 per-vector-scaled, recommended).

Each slot stores a `f32` per-vector scale (4 bytes) plus one `i8` per dimension (INT8) or two nibbles per byte (INT4). For unit-normalized vectors, INT8 quantization error is ~8√ó10‚Åª‚Åµ per dimension ‚Äî far below the score gaps between true nearest neighbors at any d.

**Disk impact at 100k vectors (bits=4, total on-disk including codes):**

| d    | rerank=False | int8 (default) | int4 | f16  | f32  |
|------|-------------|---------------|------|------|------|
| 200  | ~11 MB      | ~31 MB        | ~21 MB | ~51 MB | ~91 MB |
| 768  | ~40 MB      | ~117 MB       | ~79 MB | ~194 MB | ~347 MB |
| 1536 | ~84 MB      | ~231 MB       | ~161 MB | ~391 MB | ~698 MB |

**Recall impact (brute-force, bits=4):**

At d=200, R@1 improvement from rerank is modest (~+5‚Äì8 pp) because MSE codes already capture most of the signal. At d‚â•1536, the improvement is +15‚Äì25 pp R@1.

**`rerank_factor`** (search time, default 10 brute / 20 ANN):

The number of candidates pre-selected by the quantized pass before exact re-scoring. Higher factor = better recall at higher latency. Industry convention (Qdrant `oversampling`, LanceDB `refine_factor`):
- `rerank_factor=5`: fast, slight recall loss at top_k=1
- `rerank_factor=10`: default, good balance
- `rerank_factor=20‚Äì50`: maximum recall, use when latency is not critical

Default behavior note:
- The default oversampling is intentionally fixed (`10` brute-force / `20` ANN). It is not auto-adapted by dimension unless you pass `rerank_factor` explicitly.

```python
# High-recall, accepts 5√ó latency overhead
results = db.search(query, top_k=10, rerank_factor=50)

# Fast re-rank, accepts minor recall drop
results = db.search(query, top_k=10, rerank_factor=5)
```

---

### `quantizer_type` ‚Äî auto, `"dense"`, or `"srht"`

Selects the rotation applied before Lloyd-Max quantization.

| quantizer_type | Rotation | Ingest cost | Recall | When to use |
|----------------|----------|-------------|--------|-------------|
| `None` | Auto: `"dense"` for d<1024, `"srht"` for d>=1024 | dimension-aware | default | Let the library pick the v0.9 crossover |
| `"dense"` / `"exact"` | Haar QR | O(d¬≤) | strongest low-d / no padding | d < 1024, or when you want paper-faithful QR |
| `"srht"` | SRHT (Hadamard) | O(d log d) | equal/slightly better in high-d public benches | d >= 1024 with high throughput or latency requirements |

**Rule:** Leave `quantizer_type=None` unless you need a reproducible mode choice. Pin `"dense"` for low-dimensional or no-padding storage; pin `"srht"` for high-dimensional ingest/search speed.

---

### `_use_ann` ‚Äî brute-force vs HNSW

| `_use_ann` | When active | p50 latency | Recall penalty |
|-----------|-------------|-------------|----------------|
| `False` / `None` (auto, no index) | always | O(N) | none |
| `True` (requires `create_index()`) | N ‚â• 10k | O(log N) | ‚àí2 to ‚àí8 pp R@1 |

The auto planner (`_use_ann=None`) enables HNSW when:
- An index exists
- N ‚â• 10,000
- Unindexed delta ‚â§ 20% of corpus

Call `db.create_index()` after bulk load to build the HNSW graph.

## Recommended Presets

### Preset 1 ‚Äî Max recall, high-dimensional (d ‚â• 1536)

```python
db = Database.open(path, dimension=1536,
    bits=4, rerank=True, fast_mode=False,
    quantizer_type="dense")  # pin dense for maximum-recall QR; rerank_precision=None ‚Üí int8
# ~149 MB for 100k vectors; R@1 ‚âà 0.94‚Äì0.96 (DBpedia-1536)
```

### Preset 2 ‚Äî Max recall, low-dimensional (d ‚â§ 768)

```python
db = Database.open(path, dimension=768,
    bits=4, rerank=True, fast_mode=True,
    quantizer_type=None)  # rerank_precision=None ‚Üí int8
# ~74 MB for 100k vectors; R@1 ‚âà 0.96‚Äì0.99 (arXiv-768)
```

### Preset 3 ‚Äî Default (minimum disk, fastest ingest)

```python
db = Database.open(path, dimension=DIM, bits=4)
# rerank=False, fast_mode=True (defaults) ‚Äî MSE codes only
# ~11 MB (d=200), ~40 MB (d=768), ~84 MB (d=1536) per 100k vectors
```

### Preset 4 ‚Äî Minimum disk

```python
db = Database.open(path, dimension=DIM,
    bits=2, rerank=False, fast_mode=True)
# ~6‚Äì16 MB for 100k vectors; R@1 ‚âà 0.51‚Äì0.84 (dataset-dependent)
```

### Preset 5 ‚Äî Fast ingest, high-dimensional

```python
db = Database.open(path, dimension=1536,
    bits=4, rerank=False, fast_mode=True,
    quantizer_type="srht")
# 4,000‚Äì7,000 vps vs 2,000 vps for dense at d=1536
```

### Preset 6 ‚Äî Low latency at large N

```python
db = Database.open(path, dimension=DIM,
    bits=4, rerank=True, fast_mode=True)
db.create_index(max_degree=32, ef_construction=200)
results = db.search(query, top_k=10, _use_ann=True)
# p50 < 10ms at N=100k; R@1 ~2‚Äì5 pp below brute-force
```

## Multi-vector mode (v0.8)

`MultiVectorStore` (see [`docs/MULTI_VECTOR.md`](MULTI_VECTOR.md)) is a Python-layer
wrapper over `Database`. Every parameter above (`bits`, `metric`, `quantizer_type`, ‚Ä¶)
applies identically ‚Äî the wrapper just adds doc-id-grouping sidecars on top.

Recommended starting config for ColBERTv2-style late interaction:

| Parameter | Value | Why |
|---|---|---|
| `dimension` | 96 (ColBERTv2) or 128 (ColBERT) | per-token dim of the model |
| `bits` | 4 | good recall at modest disk |
| `metric` | `"cosine"` | ColBERT vectors are unit-normalised |
| `quantizer_type` | `None` or `"dense"` | auto resolves to dense at ColBERT dimensions; pin dense for reproducibility |

The wrapper rejects `metric="l2"` at construction ‚Äî MaxSim is a dot-product
aggregation that doesn't generalise cleanly to lower-is-better metrics.

## Async executor tuning (v0.8)

`AsyncDatabase` (see [`docs/PYTHON_API.md` ‚Äî Async API](PYTHON_API.md#async-api))
takes two construction-time knobs that aren't on `Database.open`:

| Kwarg | Default | When to override |
|---|---|---|
| `executor` | None (auto-create) | Pass an external `ThreadPoolExecutor` to share a pool across multiple `AsyncDatabase` instances. External pools are not shut down on `close()`. |
| `max_workers` | `min(32, cpu_count + 4)` | Override the auto-pool size. Ignored when `executor=` is supplied. |

The default sizing matches Python 3.13's standard `ThreadPoolExecutor` heuristic.
Raise it for heavy concurrent-search workloads on machines with many cores; lower
it on memory-constrained hosts.

## Decision Flowchart

```
Start from defaults: bits=4, rerank=False, fast_mode=True, quantizer_type=None

Need better recall?
  YES ‚Üí rerank=True (+5‚Äì25 pp R@1; adds INT8 raw vectors, ~2‚Äì4√ó more disk)
      ‚Üí rerank_precision="residual_int4" if storage is tight (~31% less disk vs int8 at b=4)
      ‚Üí rerank_precision="f16" for maximum precision

Is d ‚â• 1536 AND rerank=False?
  YES ‚Üí fast_mode=False adds QJL residual (+5‚Äì10 pp R@1)
  NO  ‚Üí keep fast_mode=True (QJL noisy at d<512; redundant when rerank=True)

Is N > 100k or p50 < 10ms required?
  YES ‚Üí create_index() + _use_ann=True (or None for auto)
  NO  ‚Üí brute-force is fine

Need a fixed quantizer mode instead of dimension-aware auto?
  d < 1024 or no-padding/paper-faithful QR ‚Üí quantizer_type="dense"
  d >= 1024 and ingest/latency matter      ‚Üí quantizer_type="srht"
  otherwise                                ‚Üí keep quantizer_type=None
```

## Storage Estimation

Approximate disk usage for 100k vectors:

```
compressed_codes_MB = N √ó ceil(d / (8 / bits)) / 1e6   # b=4: d/2 bytes, b=2: d/4 bytes
raw_int8_MB         = N √ó (d + 4) / 1e6                # rerank=True, int8 (default)
raw_int4_MB         = N √ó (ceil(d/2) + 4) / 1e6        # rerank=True, int4
raw_f16_MB          = N √ó d √ó 2 / 1e6                  # rerank=True, f16
raw_f32_MB          = N √ó d √ó 4 / 1e6                  # rerank=True, f32
hnsw_graph_MB       ‚âà N √ó max_degree √ó 4 / 1e6         # default max_degree=32 ‚Üí ~12 MB/100k
wal_overhead_MB     ‚âà 5‚Äì20% of compressed_codes_MB
```

Example (d=1536, N=100k, bits=4, rerank=True, default int8):
- Compressed codes: 100k √ó 768 bytes = ~73 MB
- INT8 raw vectors: 100k √ó (1536 + 4) bytes = ~147 MB
- HNSW graph (optional): ~12 MB
- **Total: ~230‚Äì245 MB** (vs ~378‚Äì400 MB with F16)
